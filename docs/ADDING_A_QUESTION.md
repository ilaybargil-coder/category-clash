# הוספת שאלה חדשה למאגר — מדריך מלא מא׳ עד ת׳

המדריך הזה הוא חוזה העבודה להוספת שאלה או להרחבת שאלה קיימת ב־Category
Clash. המטרה היא לקבל הרבה תשובות הגיוניות, לסלוח על כתיב אנושי, ולמנוע מצב
שבו מילה שונה זוכה בנקודה בטעות.

## 1. לפני הקוד: מגדירים את חוקי השאלה

לכל שאלה חדשה יש לכתוב תחילה מדיניות קצרה:

1. **מה נכנס?** הגדרה חיובית וברורה.
2. **מה לא נכנס?** לפחות שלוש דוגמאות גבוליות.
3. **רמת הפירוט:** האם גם משפחה וגם תת־סוג מקבלים נקודות נפרדות?
4. **זמן ומקום:** האם הרשימה היא ישראלית, עולמית, היסטורית או עדכנית?
5. **שמות מסחריים:** האם מותגים מתקבלים?
6. **שפה:** אילו תעתיקים באנגלית או בערבית מתקבלים?

דוגמה עבור `כתבו שמות של קינוחים`:

- נכנסים: עוגות, מאפים מתוקים, פודינגים, ממתקים וקינוחים קפואים שמקובל
  להגיש כמנה מתוקה.
- לא נכנסים: מנה מלוחה, חומר גלם לבדו, או שם מסעדה.
- מאפה מתוק נפוץ כמו קרואסון מתקבל במדיניות המשחק הנדיבה, גם אם אפשר לאכול
  אותו בארוחת בוקר.
- וריאציות של אותה מנה יכולות להיות aliases או קבוצה סמנטית, לפי השאלה אם
  הוגן לקבל עליהן שתי נקודות.

בלי מדיניות כזאת אי אפשר להכריע באופן עקבי במקרים גבוליים.

## 2. מבנה הנתונים

כל תשובה היא tuple בצורה הבאה:

```python
("תשובה ראשית", ["כינוי", "כתיב חלופי", "שגיאה נפוצה"], "semantic-group")
```

### תשובה ראשית — canonical

השם שהמשחק מציג לאחר שהבין את המשתמש. הוא צריך להיות:

- השם הנפוץ והברור בעברית;
- ביחיד, אלא אם הצורה המקובלת היא ברבים;
- ללא הסבר בסוגריים;
- יציב לאורך זמן.

### Alias

צורה חלופית שממופה לאותה תשובה ראשית. Alias מתאים עבור:

- תעתיק חלופי: `קוראסון` ← `קרואסון`;
- שגיאת כתיב נפוצה ומכוונת: `מילפה` ← `מילפיי`;
- יחיד ורבים: `קרואסונים` ← `קרואסון`;
- שם לועזי נפוץ: `croissant` ← `קרואסון`;
- שם מקומי או שם נוסף אמיתי: `קרמשניט` ← `מילפיי`.

Alias אינו מתאים כאשר שתי הצורות הן תשובות שונות שיכולות לזכות בנקודות
נפרדות.

### Semantic group

קבוצה סמנטית אומרת ששתי תשובות מוכרות, אבל נחשבות אותה תשובה לצורכי ניקוד.
לדוגמה, אם מדיניות השאלה מחליטה שכל וריאציות עוגת הגבינה הן נקודה אחת:

```python
("עוגת גבינה", [], "cheesecake")
("עוגת גבינה באסקית", ["צ'יזקייק באסקי"], "cheesecake")
```

אין ליצור קבוצה רק מפני ששתי מילים קשורות. `זהב` ו־`אוצר`, למשל, יכולים
להיות אותה משמעות בשאלה מסוימת ושתי תשובות שונות בשאלה אחרת.

## 3. מחקר תוכן

לכל שאלה יש להשתמש בלפחות שני סוגי מקורות:

1. מקור רשמי או מקצועי: משרד ממשלתי, התאחדות, מוסד מדעי או גוף מקצועי.
2. מקור רשימתי רחב שמסייע למצוא שמות נפוצים וחוסרים.
3. תשובות אמת שנדחו במשחק, כשהן קיימות.

כל מקור נשמר לצד השאלה. אסור להוסיף רשימה שנוצרה בידי AI ללא אימות.

### סדר העבודה במחקר

1. מוציאים את כל התשובות וה־aliases הקיימים.
2. אוספים רשימת מועמדים מהמקורות.
3. מסירים מועמדים מחוץ למדיניות השאלה.
4. מחלקים כל מועמד ל־canonical, alias, semantic group או פסול.
5. בודקים וריאציות כתיב בעברית ובאנגלית.
6. מחפשים התנגשויות בתוך השאלה הנוכחית.

## 4. כמה תשובות צריך?

אין יעד עיוור של 200 תשובות לכל שאלה. למדינות באירופה יש מספר סופי, בעוד
למקצועות או לקינוחים יש מאות אפשרויות. היעד הוא:

- כיסוי מלא ככל האפשר בקטגוריה סגורה;
- לפחות 50 צורות מדויקות בשאלת בטא פתוחה;
- 100–200 צורות מדויקות בשאלה רחבה ונפוצה;
- המשך הרחבה לפי תשובות שנדחו בפועל.

איכות קודמת לכמות: 200 מילים שגויות הופכות את המשחק לפחות הוגן.

## 5. איפה מוסיפים את הדאטה?

### הרחבת שאלה קיימת

תוספות מחזוריות נכנסות אל:

```text
backend/app/question_bank_continuous.py
```

- תשובה חדשה: `CURATED_ANSWER_ADDITIONS`.
- alias לתשובה קיימת: `CURATED_ALIAS_ADDITIONS`.
- מקורות: `CURATION_SOURCES`.

### שאלה חדשה לחלוטין

1. מוסיפים את השאלה וגרעין התשובות ל־`backend/app/question_bank.py`, בתוך
   `ADDITIONAL_QUESTIONS`.
2. מוסיפים לה הרחבות ומקורות בקובץ הדאטה המתאים.
3. אין צורך ב־Alembic כל עוד מבנה הטבלאות לא השתנה; ה־seed יוצר את השאלה.
4. מעדכנים את בדיקות הכיסוי כך שגם השאלה החדשה חייבת מקורות.

תבנית:

```python
{
    "text": "כתבו ...",
    "answers": [
        ("תשובה ראשית", ["כתיב חלופי"], None),
        ("תשובה נוספת", [], "optional-group"),
    ],
}
```

## 6. חוקים להשלמה אוטומטית ולשגיאות כתיב

ה־resolver עובד לפי הסדר:

1. התאמה מדויקת ל־canonical או alias;
2. ניסיון ללא ה״א הידיעה;
3. השלמת prefix רק אם כל ההתאמות מובילות לאותה תשובה;
4. fuzzy שמרני לשגיאת כתיב;
5. דחייה אם התוצאה עמומה.

לכן:

- `מילפ` יכול להשלים ל־`מילפיי`;
- `כבל h` יכול להשלים ל־`כבל HDMI`;
- קלט קצר או קלט שמצביע על שתי תשובות אינו מושלם אוטומטית;
- לא מורידים את ספי ה־fuzzy כדי לפתור חוסר בדאטה — מוסיפים alias מפורש.

כל חוק הוא **פר שאלה**. Alias מקטגוריית קינוחים לעולם אינו משפיע על שאלה
אחרת.

## 7. בדיקות שחובה להוסיף

לכל שינוי תוכן יש לפחות:

1. טסט לתשובה הראשית;
2. טסט לכתיב החלופי שבגללו נוצר השינוי;
3. טסט להשלמה ייחודית, אם רלוונטי;
4. audit שמוכיח שאין שתי תשובות עם אותה צורה מנורמלת;
5. טסט שמוודא שלשאלה יש מקורות.

דוגמה:

```python
def test_mille_feuille_common_typo_is_accepted():
    result = validator.check("מילפה")
    assert result.status == AnswerStatus.VALID
    assert result.entry.canonical == "מילפיי"
```

## 8. הרצה מקומית

מתיקיית `backend`:

```bash
python -m scripts.audit_question_bank
ruff check app tests scripts
ruff format --check app tests scripts
mypy app
pytest -q
```

ה־audit חייב להסתיים ללא collisions. אזהרת singleton group דורשת בדיקה, אך
אינה בהכרח תקלה.

## 9. Seed ו־Supabase

`python -m app.seed` הוא אידמפוטנטי:

- מוסיף שאלות ותשובות חסרות;
- מוסיף aliases חסרים;
- אינו מכפיל נתונים קיימים;
- אינו מוחק היסטוריית משחקים.

מחיקה או שינוי שם של תשובה קיימת דורשים מנגנון rename/deactivation ב־seed,
משום שתשובות היסטוריות עשויות להפנות אליה באמצעות foreign key.

אין להכניס `DATABASE_URL`, מפתח service-role או סיסמה ל־Git.

## 10. Pull request ופריסה

1. עובדים בענף/worktree נפרד.
2. מעלים רק קבצי דאטה, מקורות וטסטים השייכים לבאץ׳.
3. פותחים draft PR.
4. CI מריץ seed על PostgreSQL זמני, audit, lint, type-check וכל הטסטים.
5. אדם בודק את ההחלטות הסמנטיות וממזג.
6. Render פורס את `main` ומריץ migrations + seed מול Supabase.
7. מאמתים `/health` ו־`/ready`.

## 11. לולאת השיפור הקבועה

המשחק כבר שומר כל ניסיון ב־`submitted_answers`. אחת לשבוע יש להריץ:

```bash
python -m scripts.report_rejected_answers --min-count 2 --limit 100
```

הדוח מדרג מילים שנדחו לפי שכיחות ומספר שחקנים שונים. כל מועמד מסווג כאחד:

- `NEW_CANONICAL` — תשובה חדשה;
- `ALIAS` — ניסוח של תשובה קיימת;
- `SEMANTIC_GROUP` — תשובה חוקית אך כפולה סמנטית;
- `INVALID` — אינה עונה על השאלה;
- `DEFER` — עמומה ודורשת בדיקה נוספת.

סוכן יכול לחקור ולהכין draft PR. הוא לעולם לא מאשר את עצמו, לא משנה ספי
fuzzy ולא כותב ישירות ל־Supabase.
